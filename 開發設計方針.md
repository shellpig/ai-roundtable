## 常數與環境變數（server.py 頂部，比照 `PUBLIC_MODE` 區塊）

```python
DEVMODE = os.environ.get("AI_ROUNDTABLE_DEVMODE") == "1"
DEV_CALL_TIMEOUT = int(os.environ.get("AI_ROUNDTABLE_DEV_TIMEOUT", "3600"))   # 開發模式單棒上限（秒）
DEV_MAX_TURNS = int(os.environ.get("AI_ROUNDTABLE_DEV_MAX_TURNS", "40"))      # 整場總棒數上限
DEV_MAX_ATTEMPTS = 3            # 單任務重試上限（規格 §5）
DEV_BRANCH_PREFIX = "roundtable/dev-"
TASKS_FILE = DATA / "tasks.json"
MEETING_SUMMARY_FILE = DATA / "meeting_summary.json"
DEVLOG_DIR = DATA / "devlogs"
RATE_LIMIT_MARKERS = ("rate limit", "usage limit", "429", "quota", "usage_limit")  # 小寫比對
```

- `main()` 開頭：`DEVMODE and PUBLIC_MODE` → 印出互斥原因後 `sys.exit(1)`。
- `main()` 綁定：DEVMODE 時跳過 `_tailscale_ip()` 綁定，只綁 `127.0.0.1`。
- 邀請停用：`/api/token/generate`、`/api/token/verify` 在 DEVMODE 下回 403；前端以 state 旗標隱藏邀請鈕。

## 角色設定（settings.json 擴充）

`_default_settings()` / `_coerce_settings()` 增加：

```json
"dev_roles": {"controller": "claude", "implementer": "agy", "verifier": "codex"}
```

驗證值必須是 `PARTICIPANTS` 的 key 且三角色互異，非法時回落預設。合法值依使用者設定原樣保留；伺服器不依模型名稱推測能力，也不自動改派。v1 不做 UI 編輯，只在 `/api/state` 的 dev payload 回傳供顯示。

預設映射是正式安全驗收基準。非預設映射屬使用者自控：可能無法寫入或不再滿足唯讀假設，`/api/state` 另回傳 `roles_custom: true`，前端顯示能力與安全性警告。未來新增模型只需登錄 participant／adapter；是否適任某角色仍由使用者決定。

## Adapter 參數變更（僅 DEVMODE 且該棒屬管線時生效）

下表描述**預設映射**。管線一律把 `dev_role` 傳給被選席位的 adapter，但 v1 不承諾任意 adapter 都能實作所有角色；使用者採用非預設映射前須自行確認。

| 席位 | 討論版（不變） | 開發管線棒 |
|---|---|---|
| claude（主控） | `--allowedTools Read,Glob,Grep --add-dir DATA` | 同左（仍排除 web 工具），timeout 改 `DEV_CALL_TIMEOUT` |
| agy（實作） | `--sandbox` | 移除 `--sandbox`，改 `--dangerously-skip-permissions`；timeout `DEV_CALL_TIMEOUT` |
| codex（驗證） | `--sandbox read-only` | 同左；timeout `DEV_CALL_TIMEOUT` |

實作方式：`_call_codex/_call_agy/_call_claude` 增加關鍵字參數（如 `dev_role=None, timeout=CALL_TIMEOUT`），由管線 worker 傳入；一般 `ask()` 路徑完全不帶，確保討論行為零變化。

### D4 結構化結果、usage 與 session

開發管線內部的 adapter 統一回傳 `AdapterResult`；一般討論路徑仍只取 `text`，不得改變既有 `ask()` 對外行為：

```json
{
  "text": "模型回覆本文",
  "usage": {
    "source": "cli_json | unavailable",
    "input_tokens": null,
    "cached_input_tokens": null,
    "output_tokens": null,
    "total_tokens": null,
    "cost_usd": null
  },
  "session": {
    "id": null,
    "resumed": false,
    "resume_failed": false
  }
}
```

- Codex 管線棒使用 `exec --json` 逐行解析事件：回覆本文取最後一個 agent message，usage 取 `turn.completed.usage`，明確 session ID 取 `thread.started.thread_id`。需要續用時使用 `exec resume <ID> --json`；驗證角色維持全新 session。不得以人類可讀 footer 作主要資料源。
- Claude 管線棒使用 `--output-format json`，從 JSON 取 `result`、`usage`、`total_cost_usd` 與 `session_id`；續用時只使用 `--resume <ID>`。缺少的 token 欄位保持 null。
- agy D4 不做 token usage；若無法從機器可讀輸出取得明確 conversation ID，session 標記 unavailable 並維持全新 subprocess，不得使用隱式 `--continue`。
- JSON 解析失敗但 CLI 成功時，該棒視為 `seat_error` 並保留原 stdout/stderr 供診斷，不能把 usage 猜成 0。resume 明確回報 session 無效時只重試一次全新 session，並將 `resume_failed` 寫入 devlog。

## 管線狀態（模組級，比照 `_discussion`）

```python
_dev = {
    "active": False, "paused": False, "pause_reason": "",  # pause_reason: manual/interject/rate_limit/tamper/parse_fail/seat_error/turn_cap/crash/ask_host

```

執行緒模型比照 `run_discussion()`：單一背景 thread 跑整條管線，人類插話偵測比照 `_host_interrupted()`（記 baseline 訊息數，每棒結束後檢查）。

## tasks.json schema（v2，D4）

```json
{
  "version": 2,
  "status": "running | paused | done",
  "pause_reason": "",
  "branch": "roundtable/dev-20260710-153000",
  "project_dir": "C:\\path\\to\\project",
  "session_goal": "開場目標原文",
  "turn_count": 7,
  "message_watermark": 5,
  "dispatched": true,
  "sessions": {
    "controller": {"provider": "claude", "session_id": "...", "task_id": null, "branch": "roundtable/dev-..."},
    "implementer": {"provider": "agy", "session_id": null, "task_id": 1, "branch": "roundtable/dev-..."}
  },
  "usage": {
    "last": {},
    "by_provider": {"codex": {}, "claude": {}},
    "known_total_tokens": 0,
    "incomplete": true
  },
  "updated_at": "2026-07-10 15:30:00",
  "tasks": [
    {
      "id": 1, "title": "...", "files": ["..."], "acceptance": ["..."],
      "status": "pending | in_progress | done | blocked",
      "attempts": 0, "arbitrated": false,
      "base_commit": "9f8e7d6", "commits": ["a1b2c3d"],
      "last_verdict": "不通過—token 無過期時間",
      "last_failure_fingerprint": "verify:token 無過期時間",
      "consecutive_same_failures": 1
    }
  ],
  "handoff": "收尾棒摘要原文"
}
```

寫入一律走 `_write_atomic(path, text)`：同目錄暫存檔 + `os.replace`。伺服器啟動時若 `status == "running"` → 改寫為 `paused` / `pause_reason = "crash"`。

載入 version 1 時做單向、可重複的記憶體 migration：補入空 `sessions`、usage 初始值、各 task 的 `base_commit`／failure 欄位後，以 version 2 原子寫回；既有未完成管線不得因升版遺失 branch、commit、attempts 或 message watermark。`tasks.json` 仍是斷點續作唯一事實來源；`meeting_summary.json` 只保存語意摘要。

- `message_watermark`：暫停時記當下最新人類訊息編號（`_dev_pause_state` 更新）；續作時水位之後有新的人類發言（含仲裁「詢問」後主持人的拍板）→ 管線第一棒先跑消化棒（規格 §5）。
- `dispatched`：新管線建立時為 false，主控成功拆出首份任務清單後改為 true；用來區分「尚未拆任務的空清單」與「消化棒刪除全部未完成任務後的空清單」，後者必須進入 handoff，不得重新 dispatch。
- `base_commit`：任務第一次從 pending 進入 in_progress 前記錄當時 HEAD；所有該任務重試沿用同一值。任務驗證使用 `base_commit...HEAD`，最終整合驗收才使用 `main...HEAD`。
- `sessions`：只保存 controller 與目前 implementer 可續用的明確 ID；verifier 每棒全新，不保存為下一棒 resume 來源。provider、角色、project_dir、branch 或 implementer task_id 不符時丟棄 ID。
- `usage`：只累加非 null 的 Codex／Claude token 欄位；`incomplete` 表示至少一棒或 adapter 無可用 usage。`last` 供 devBar 顯示最近一棒，`by_provider` 與 `known_total_tokens` 供累計顯示；不得掃描歷史 log 即時計算。
- 續作語意（規格 §5）：`turn_count` 於每次「▶」續作歸零重計（重新授權一輪預算）；髒 working tree 且當前分支為管線分支時，伺服器先 commit 殘留變更（訊息 `[roundtable] 任務{id} 中斷殘留變更`）再續跑。

## meeting_summary.json schema（D4）

```json
{
  "version": 1,
  "source_message_watermark": 12,
  "goal": "完成登入 API",
  "decisions": ["先做後端，不做 UI"],
  "non_goals": ["OAuth"],
  "global_constraints": ["不得新增第三方套件"],
  "acceptance_criteria": ["既有測試維持通過"],
  "open_questions": [],
  "updated_at": "2026-07-10 15:30:00"
}
```

主控拆任務／消化棒在同一份協議輸出中同時回傳 `meeting_summary` 與 `tasks`；伺服器先完整驗證兩者，任一不合法都不得部分接受。摘要由伺服器補 `version`／`updated_at` 後原子寫入。`source_message_watermark` 不得倒退；摘要不存在、JSON 損壞或水位大於現有訊息數時，下一次主控棒退回完整逐字稿重建。

摘要加入 `_data_hashes` 與 `_unique_archive_paths` 管理範圍。主控 prompt 讀完整摘要及水位後訊息；實作／驗證 prompt 由伺服器只投影與當前任務相關的 `decisions`、`global_constraints`、`acceptance_criteria`，不得提供摘要路徑或完整內容。

## 協議常數與解析（規格 §6 的逐字契約）

格式字串定義為模組常數，prompt 模板與解析器共用；不得在兩處各寫一份字面值。

- `_parse_tasks(text, *, allow_empty=False, previous_summary=None)`：取**最後一個** ```json 圍欄 → `json.loads` → 一次嚴格驗證必要的 `meeting_summary` 與 `tasks`。摘要驗證依上一節；tasks 的 id 必須是非 bool 的正整數且唯一，title 非空，files 為字串陣列，acceptance 為至少一個非空字串。拆任務使用 `allow_empty=False`，消化棒使用 `True`。消化棒以 id 對齊，`status == "done"` 的任務保留原紀錄；新清單缺少的未完成任務視為刪除。
- `_parse_verdict(text)`：最後一個非空行必須完全等於 `驗證結果：通過`，或以 `驗證結果：不通過—` 開頭且後方原因非空。
- `_parse_arbitration(text)`：最後一個非空行以三種仲裁前綴之一開頭，且 `—` 後內容必須非空。
- 失敗處理統一包裝：`_protocol_call(...)` 跑一次 → 失敗附「上次輸出不符格式，請重新輸出」重跑一次 → 再失敗 `pause_reason = "parse_fail"`。

## Git 整合（全部由伺服器執行，`subprocess.run` 於 `_project_dir`）

- `_git(*args)` 小工具：回傳 CompletedProcess，統一 UTF-8。
- 前置檢查：`git rev-parse --is-inside-work-tree`；`git status --porcelain` 非空即拒。
- 開分支：`git checkout -b {DEV_BRANCH_PREFIX}{_stamp()}`；續作改 `git checkout {branch}`（tasks.json 記錄值）。
- 任務首次進入 in_progress：以 `git rev-parse HEAD` 記錄 `base_commit`，後續重試不得覆寫。
- 實作棒後：`git add -A` → `git status --porcelain` 空 = 無變更 → 該次嘗試記失敗（feedback「無任何檔案變更」）不進驗證；有變更 → `git commit -m "[roundtable] 任務{id} {seat} 第{attempts}次: {title}"`，`git rev-parse --short HEAD` 記入 `commits`。
- 任務驗證棒指令內嵌：任務簡報 + `git diff --name-status {base_commit}...HEAD` + `git diff --stat {base_commit}...HEAD` + 適用限制 + 越界即不通過要求；不提供歷史 commit 清單。完整 patch 由驗證席自行以同一 range 的唯讀 git 指令查。
- 收尾前新增一次整合驗證棒，以管線開始時的 main commit 到 HEAD（等價於本管線的 `main...HEAD`）檢查全體最終 net diff；通過才進 handoff，失敗走一般驗證失敗處理。

## 相同失敗提前煞車（D4）

`_failure_fingerprint(source, reason)` 產生可重現字串：source 使用 `implement_no_change`／`implement_error`／`verify` 等固定分類；reason 先做 Unicode NFKC、轉小寫、首尾去空白並壓縮連續空白。D4 不做語意相似判斷，避免把不同原因誤合併。

每次任務失敗呼叫 `_record_task_failure(task, source, reason)`：指紋等於上一個時 `consecutive_same_failures += 1`，否則覆寫指紋並設為 1。達 2 時先持久化 tasks.json，再以 `pause_reason = "repeated_failure"` 暫停，禁止第三棒。新的實作 commit 不重置；只有消化／仲裁實質改變 title、files、acceptance 或重派指示時清空，或下一個指紋不同時自然重置。限流與協議解析失敗沿用各自的立即暫停路徑，不納入此計數。

## 竄改偵測

`_data_hashes = {}`：`append_message()`、`_write_atomic()` 等伺服器自身寫入點之後更新 SHA-256（對象：`TRANSCRIPT`、`MD_MIRROR`、`TASKS_FILE`、`MEETING_SUMMARY_FILE`）。`_trusted_tasks` 與 `_trusted_meeting_summary` 同時保留最近一次由伺服器成功寫入／啟動時載入的深拷貝。每棒開始前重算雜湊；不一致時以 trusted snapshot 寫入 paused/tamper，絕不可把遭竄改的磁碟資料讀回後重新保存成可信狀態，並在逐字稿加入警告。

## 限流偵測

席位呼叫拋錯或非零 exit 時，將 stdout+stderr 轉小寫比對 `RATE_LIMIT_MARKERS`；命中 → 不計 attempts、`pause_reason = "rate_limit"`。未命中維持原失敗路徑。

## 稽核日誌

每棒寫 `DEVLOG_DIR / f"{_stamp()}-task{id}-{seat}-{stage}.log"`：指令 args、完整 instruction、stdout、stderr、exit code、耗時，並追加單行 `metadata_json=<JSON>`，內容含正規化 usage 與 session metadata。管線棒必寫；討論版不寫（維持現狀）。usage 同時在該棒結束時更新 tasks.json 累計，`/api/state` 不掃描歷史 log。

## 上下文裁剪（指令模板，新增 `_dev_instruction(stage, ...)` 系列）

| 棒 | 給的上下文 |
|---|---|
| 首次拆任務（主控） | 完整逐字稿路徑 + 專案目錄 + 現行 tasks.json；同棒產生摘要 |
| 仲裁／消化／收尾（主控） | 完整 meeting summary + 摘要水位後的新訊息 + 現行 tasks.json；摘要失效才回退完整逐字稿 |
| 實作 | 專案目錄 + 規格文件指引 + 當前任務簡報 + 相關決議／全域限制 + 上次驗證意見。不讀逐字稿或完整摘要 |
| 驗證 | 任務簡報 + 適用限制 + `base_commit...HEAD` 最終 net diff。不讀逐字稿、完整摘要或歷史 commit 清單 |

## API 與 UI

- `POST /api/dev/start`（HOST）：前置檢查 → 啟動或續跑管線。`POST /api/dev/pause`（HOST）：請求暫停（當前棒跑完生效）。非 DEVMODE 一律 403。
- 管線 active 或 paused 未收尾時（`_dev_pipeline_engaged()`），`POST /api/send` 只 `append_message` 並回傳 `dev_pending: true`（`mode=discussion` 同樣被抑制）；`POST /api/ask` 回 409 帶原因。不得呼叫一般 `start_batch`／`start_discussion`。active 時由棒後檢查觸發消化；paused 時等下次 `/api/dev/start` 先消化。
- `/api/state` 增加 `"dev": {"devmode": bool, "active", "paused", "pause_reason", "stage", "current_task", "task_total", "turn_count", "branch", "roles", "roles_custom", "usage"}`；usage 回傳 `last`、`by_provider`、`known_total_tokens`、`incomplete`。
- `/api/new`：`_dev["active"]` 或 paused 未收尾時回 409 與原因（規格 §5）。會議封存時把 `tasks.json` 與 `meeting_summary.json` 一併改名封存（比照 `_unique_archive_paths`）。
- `index.html`：header 加開發模式徽章；composer 加 `#devBtn`（▶/⏸，依 `dev` payload 切換與顯示）；`renderDiscussBar` 旁增 `renderDevBar`（任務 m/n、stage、分支、暫停/警告原因、角色映射、最近一棒與本輪已知 usage）；`roles_custom` 為 true 時顯示能力與安全性由使用者確認的警告；usage 的 null／unavailable 顯示為 `usage unavailable`，不得格式化成 0；管線訊息以 `sub` 欄位顯示「任務 n · 實作」等標籤。

## 實作 phase 與完成定義

**Phase D1 — 模式進入與權限**：start_devmode.cmd、互斥檢查、只綁 loopback、邀請停用、dev_roles 載入、adapter 參數分流、UI 徽章。完成定義 = 規格 §3 全過；§4 的合法／非法角色設定與討論版參數不變通過。管線 devlog、竄改與驗證 prompt 不屬 D1。

**Phase D2 — 管線核心與安全網**：狀態機、嚴格協議解析、git 整合、tasks.json 原子寫入與 crash 降級、重試／仲裁、棒數與限流煞車、竄改偵測、稽核日誌、上下文裁剪及確定性測試。完成定義 = 預設角色的管線參數、§4 竄改／驗證 prompt、§5 的 git／重試／仲裁／上限／限流／crash／殘留 commit、§6、§7 全過，且 `tests.test_devmode` 通過；不含人類插話路由與 UI。

**Phase D3 — UI 與續作體驗**：▶/⏸ 按鈕、devBar、角色映射警告、active／paused 發言只入逐字稿、插話→消化→續跑、`/api/new` 禁用與封存整合。完成定義 = §5 的手動暫停、人類發言抑制一般 batch、訊息水位／消化續作、新會議與封存項目，加上 §8 全過；相關 unittest 與手動 UI 驗收皆通過。

**Phase D4 — 上下文、Session 與用量可觀測性**：per-task final net diff 與收尾整合 diff、meeting summary 建立／裁剪／雜湊／封存、相同失敗第二次提前暫停、Codex／Claude 結構化 usage、依角色 session 政策、resume fallback，以及 devBar 每棒／累計顯示。完成定義 = 規格 §4 的 final diff、§5 的 repeated failure／session、§6 的摘要協議、§7 的摘要／usage、§8 的 usage UI 全過；`tests.test_devmode` D4 fake gate 全過，並完成一次不列入日常 gate 的 Codex／Claude 真 CLI session nonce smoke。

## 邊界

- 不動討論模式任何行為與席位參數；不動三個安全 commit 的既有防線（Host 白名單、Claude 無 web 工具、agy 討論版無 bypass）。
- 不引入第三方套件；正式 runtime 維持單檔 server.py + 單檔 index.html，測試獨立放在 `tests/`。
