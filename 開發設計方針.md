## 常數與環境變數（server.py 頂部，比照 `PUBLIC_MODE` 區塊）

```python
DEVMODE = os.environ.get("AI_ROUNDTABLE_DEVMODE") == "1"
DEV_CALL_TIMEOUT = int(os.environ.get("AI_ROUNDTABLE_DEV_TIMEOUT", "3600"))   # 開發模式單棒上限（秒）
DEV_MAX_TURNS = int(os.environ.get("AI_ROUNDTABLE_DEV_MAX_TURNS", "40"))      # 整場總棒數上限
DEV_MAX_ATTEMPTS = 3            # 單任務重試上限（規格 §5）
DEV_BRANCH_PREFIX = "roundtable/dev-"
TASKS_FILE = DATA / "tasks.json"
DEVLOG_DIR = DATA / "devlogs"
RATE_LIMIT_MARKERS = ("rate limit", "usage limit", "429", "quota", "usage_limit")  # 小寫比對
```

- `main()` 開頭：`DEVMODE and PUBLIC_MODE` → 印出互斥原因後 `sys.exit(1)`。
- `main()` 綁定：DEVMODE 時跳過 `_tailscale_ip()` 綁定，只綁 `127.0.0.1`。
- 邀請停用：`/api/token/generate`、`/api/token/verify` 在 DEVMODE 下回 403；前端以 state 旗標隱藏邀請鈕。

## 角色設定（settings.json 擴充）

`_default_settings()` / `_coerce_settings()` 增加：

```json
"dev_roles": {"controller": "claude", "implementer": "agy", "verifier": "codex"}
```

驗證值必須是 `PARTICIPANTS` 的 key 且三角色互異，非法時回落預設。v1 不做 UI 編輯，只在 `/api/state` 的 dev payload 回傳供顯示。

## Adapter 參數變更（僅 DEVMODE 且該棒屬管線時生效）

| 席位 | 討論版（不變） | 開發管線棒 |
|---|---|---|
| claude（主控） | `--allowedTools Read,Glob,Grep --add-dir DATA` | 同左（仍排除 web 工具），timeout 改 `DEV_CALL_TIMEOUT` |
| agy（實作） | `--sandbox` | 移除 `--sandbox`，改 `--dangerously-skip-permissions`；timeout `DEV_CALL_TIMEOUT` |
| codex（驗證） | `--sandbox read-only` | 同左；timeout `DEV_CALL_TIMEOUT` |

實作方式：`_call_codex/_call_agy/_call_claude` 增加關鍵字參數（如 `dev_role=None, timeout=CALL_TIMEOUT`），由管線 worker 傳入；一般 `ask()` 路徑完全不帶，確保討論行為零變化。

## 管線狀態（模組級，比照 `_discussion`）

```python
_dev = {
    "active": False, "paused": False, "pause_reason": "",  # pause_reason: manual/interject/rate_limit/tamper/parse_fail/turn_cap/crash
    "stage": "", "current_task": None, "turn_count": 0, "branch": "",
}
```

執行緒模型比照 `run_discussion()`：單一背景 thread 跑整條管線，人類插話偵測比照 `_host_interrupted()`（記 baseline 訊息數，每棒結束後檢查）。

## tasks.json schema（v1）

```json
{
  "version": 1,
  "status": "running | paused | done",
  "pause_reason": "",
  "branch": "roundtable/dev-20260710-153000",
  "project_dir": "C:\\path\\to\\project",
  "session_goal": "開場目標原文",
  "turn_count": 7,
  "updated_at": "2026-07-10 15:30:00",
  "tasks": [
    {
      "id": 1, "title": "...", "files": ["..."], "acceptance": ["..."],
      "status": "pending | in_progress | done | blocked",
      "attempts": 0, "arbitrated": false,
      "commits": ["a1b2c3d"], "last_verdict": "不通過—token 無過期時間"
    }
  ],
  "handoff": "收尾棒摘要原文"
}
```

寫入一律走 `_write_atomic(path, text)`：同目錄暫存檔 + `os.replace`。伺服器啟動時若 `status == "running"` → 改寫為 `paused` / `pause_reason = "crash"`。

## 協議常數與解析（規格 §6 的逐字契約）

格式字串定義為模組常數，prompt 模板與解析器共用；不得在兩處各寫一份字面值。

- `_parse_tasks(text)`：取**最後一個** ```json 圍欄 → `json.loads` → 驗證每項含 `id/title/acceptance`。消化棒合併規則：以 id 對齊，`status == "done"` 的任務保留原紀錄不可變更；新清單缺少的未完成任務視為刪除。
- `_parse_verdict(text)`：最後一個非空行必須以 `驗證結果：通過` 或 `驗證結果：不通過—` 開頭。
- `_parse_arbitration(text)`：最後一個非空行以 `仲裁：重派—` / `仲裁：跳過—` / `仲裁：詢問—` 開頭。
- 失敗處理統一包裝：`_protocol_call(...)` 跑一次 → 失敗附「上次輸出不符格式，請重新輸出」重跑一次 → 再失敗 `pause_reason = "parse_fail"`。

## Git 整合（全部由伺服器執行，`subprocess.run` 於 `_project_dir`）

- `_git(*args)` 小工具：回傳 CompletedProcess，統一 UTF-8。
- 前置檢查：`git rev-parse --is-inside-work-tree`；`git status --porcelain` 非空即拒。
- 開分支：`git checkout -b {DEV_BRANCH_PREFIX}{_stamp()}`；續作改 `git checkout {branch}`（tasks.json 記錄值）。
- 實作棒後：`git add -A` → `git status --porcelain` 空 = 無變更 → 該次嘗試記失敗（feedback「無任何檔案變更」）不進驗證；有變更 → `git commit -m "[roundtable] 任務{id} {seat} 第{attempts}次: {title}"`，`git rev-parse --short HEAD` 記入 `commits`。
- 驗證棒指令內嵌：任務簡報 + `git show --stat` 各 commit 的輸出 + 越界即不通過的要求；細節 diff 由驗證席自行以唯讀 git 指令查。

## 竄改偵測

`_data_hashes = {}`：`append_message()`、`_write_atomic()` 等伺服器自身寫入點之後更新 SHA-256（對象：`TRANSCRIPT`、`MD_MIRROR`、`TASKS_FILE`）。每棒開始前重算比對，不一致 → `pause_reason = "tamper"`、逐字稿加系統警告訊息。

## 限流偵測

席位呼叫拋錯或非零 exit 時，將 stdout+stderr 轉小寫比對 `RATE_LIMIT_MARKERS`；命中 → 不計 attempts、`pause_reason = "rate_limit"`。未命中維持原失敗路徑。

## 稽核日誌

每棒寫 `DEVLOG_DIR / f"{_stamp()}-task{id}-{seat}-{stage}.log"`：指令 args、完整 instruction、stdout、stderr、exit code、耗時。管線棒必寫；討論版不寫（維持現狀）。

## 上下文裁剪（指令模板，新增 `_dev_instruction(stage, ...)` 系列）

| 棒 | 給的上下文 |
|---|---|
| 拆任務／仲裁／消化／收尾（主控） | 完整逐字稿路徑 + 專案目錄 + 現行 tasks.json 內容 |
| 實作 | 專案目錄 + 規格文件指引 + 當前任務簡報 + 上次驗證意見（如有）。不讀逐字稿 |
| 驗證 | 任務簡報 + commit 清單與 stat + 驗收條件。不讀逐字稿 |

## API 與 UI

- `POST /api/dev/start`（HOST）：前置檢查 → 啟動或續跑管線。`POST /api/dev/pause`（HOST）：請求暫停（當前棒跑完生效）。非 DEVMODE 一律 403。
- `/api/state` 增加 `"dev": {"devmode": bool, "active", "paused", "pause_reason", "stage", "current_task", "task_total", "turn_count", "branch", "roles"}`。
- `/api/new`：`_dev["active"]` 或 paused 未收尾時回 409 與原因（規格 §5）。會議封存時把 `tasks.json` 一併改名封存（比照 `_unique_archive_paths`）。
- `index.html`：header 加開發模式徽章；composer 加 `#devBtn`（▶/⏸，依 `dev` payload 切換與顯示）；`renderDiscussBar` 旁增 `renderDevBar`（任務 m/n、stage、分支、暫停/警告原因）；管線訊息以 `sub` 欄位顯示「任務 n · 實作」等標籤（`append_message` 既有 `sub` 參數）。

## 實作 phase 與完成定義

**Phase D1 — 模式進入與權限**：start_devmode.cmd、互斥檢查、只綁 loopback、邀請停用、dev_roles 設定、adapter 參數分流、UI 徽章。
完成定義 = 規格 §3、§4 驗收全過（§4 竄改項除外）。

**Phase D2 — 管線核心與安全網**：狀態機、協議常數與解析、git 整合、tasks.json 原子寫入與 crash 降級、全域煞車、限流偵測、竄改偵測、稽核日誌、上下文裁剪模板。
完成定義 = 規格 §5、§6、§7 驗收全過 + §4 竄改項。

**Phase D3 — UI 與續作體驗**：▶/⏸ 按鈕、devBar、插話→消化→續跑、/api/new 禁用邏輯、封存整合。
完成定義 = 規格 §8 驗收全過 + §5 的插話/續作項。

## 邊界

- 不動討論模式任何行為與席位參數；不動三個安全 commit 的既有防線（Host 白名單、Claude 無 web 工具、agy 討論版無 bypass）。
- 不引入第三方套件；維持單檔 server.py + 單檔 index.html。
