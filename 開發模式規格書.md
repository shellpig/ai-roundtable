# ai-roundtable 開發模式規格書

> 意圖層文件：開發模式「什麼狀態必須為真」與各段驗收條件。不寫操作步驟、不寫函式簽名。
> **角色歸屬**：主持人（使用者）與規劃者維護；實作者不得修改，只能列建議。
>
> | 寫什麼 | 去哪 |
> |---|---|
> | 行為規格、狀態機、協議、安全模型、驗收意圖 | **本檔** |
> | 檔案清單、函式修改點、資料 schema、CLI 參數、實作 phase | `開發設計方針.md` |
> | 手動驗收步驟（click-by-click）、測試建置方式 | `測試指南.md` |

## 1. 目標與非目標

**目標**：在既有討論圓桌之上，新增一個「開發模式」：AI 席位依角色（主控／實作／驗證）組成固定管線，對目標專案做可讀可寫的自動開發，人類只需開場交代目標、中途可插話、結束時拿到可續作的交接狀態。

**非目標（v1 不做）**：

- 主控動態調度（自由決定下一棒是誰）——v1 是伺服器寫死的固定管線。
- 席位間 session 記憶延續——每棒都是全新 subprocess，靠讀檔重建上下文。
- 「中止」動作（砍掉執行中的 subprocess）——v1 只有「暫停」（跑完當前棒停下）。單席位取消既有功能不受影響。
- 任務進度面板 UI——v1 以逐字稿與 `data/tasks.json` 為進度來源。
- 多人 guest 參與開發會議。

## 2. 使用流程（敘事）

1. 使用者以 `start_devmode.cmd` 啟動，選定專案目錄。畫面出現「開發模式」標示與「▶ 開始執行」按鈕。
2. 使用者正常發言（例：「我要做登入功能，規格在 spec.md 第 3 節，這次先完成後端 API」）。此時行為與討論圓桌完全相同：已啟用席位都會回覆，可先討論幾輪收斂方向。
3. 使用者按「▶ 開始執行」。主控席讀取完整逐字稿與規格書，拆出有序任務清單（每項含目標、涉及檔案、驗收條件），清單寫入 `data/tasks.json` 並顯示在逐字稿。
4. 管線自動逐任務執行：實作席動手 → 伺服器 commit → 驗證席驗收 → 通過則主控派下一任務；不通過則帶著驗證意見自動退回實作席重做。
5. 使用者中途發言：管線在當前棒跑完後暫停，主控消化新訊息（可能修訂任務清單），然後自動續跑。
6. 結束（任務清空、達上限、或碰到用量限制自動暫停）：狀態已在 `data/tasks.json`，下次會議按「▶ 開始執行」即可續作，主控讀狀態接棒，使用者不必重新交代。

## 3. 模式進入與安全前提

- 開發模式只能由啟動層進入：`start_devmode.cmd` 設定 `AI_ROUNDTABLE_DEVMODE=1` 後啟動伺服器。**執行期沒有任何切換模式的 API 或 UI**——可寫權限烙在啟動時組出的 CLI 參數裡，避免執行期請求（含 prompt injection）翻轉唯讀圓桌。
- **與公開模式互斥**：`AI_ROUNDTABLE_DEVMODE=1` 與 `AI_ROUNDTABLE_PUBLIC=1` 同時存在時，伺服器直接拒絕啟動並說明原因。
- 開發模式只綁 `127.0.0.1`，**不**嘗試綁 Tailscale IP。
- 開發模式停用邀請碼／guest 加入（邀請按鈕不顯示、token API 拒絕）。這是單人本機工作模式。
- 專案目錄必須是 git repo 才能啟動管線（見 §5 前置檢查）；非 git repo 仍可當討論圓桌用。

### 驗收

- [ ] 兩個環境變數同時為 1 時伺服器啟動失敗，訊息說明互斥原因。
- [ ] 開發模式下伺服器只監聽 loopback。
- [ ] 開發模式下邀請功能整條路徑（UI 按鈕與 API）皆不可用。
- [ ] 一般發言、共識討論模式、單席位取消在開發模式下行為與討論版完全一致。

## 4. 角色與權限矩陣

角色→席位的對應存於 `data/settings.json`，預設如下；v1 由設定檔調整，UI 只顯示不提供編輯。

| 角色 | 預設席位 | 專案目錄權限 | 強制機制 | 備註 |
|---|---|---|---|---|
| 主控 | Claude | 唯讀 | `--allowedTools Read,Glob,Grep`（維持排除 web 工具） | 不寫任何檔案；透過結構化輸出與伺服器溝通（§6） |
| 實作 | agy | **可寫（bypass）** | 無——`--dangerously-skip-permissions` | 唯一有筆的席位；殘餘風險見 §9 |
| 驗證 | Codex | 唯讀 | `--sandbox read-only` | 真沙箱，可執行唯讀 git 指令查 diff |

**單一寫者原則**：`data/` 目錄下所有檔案（逐字稿、`tasks.json`、settings）只由伺服器寫入。主控的任務清單與仲裁結果一律以結構化輸出交給伺服器解析落檔，主控本身不需要寫檔權限。

**越界防護（兩道，皆為偵測而非阻止）**：

1. **驗證席查 diff 越界**：每次驗證指令都包含本任務的 commit 清單與任務簡報，驗證席必須先比對 `git show --stat` 與任務範圍，凡動到與任務無關的檔案即判不通過並列出越界檔案。
2. **伺服器雜湊比對**：伺服器對自己管理的 `data/` 檔案（逐字稿 jsonl、md 鏡像、tasks.json）在每次自身寫入後記錄雜湊；每棒開始前重新計算，不一致即代表有席位繞過伺服器竄改（例如企圖對下一棒 prompt injection），立即暫停管線並在逐字稿與 UI 顯示警告。

### 驗收

- [ ] 開發模式下三席位實際啟動參數符合上表（可由稽核日誌確認）。
- [ ] 討論版（非開發模式）的席位參數完全不變，仍為現行唯讀配置。
- [ ] 手動竄改 `data/roundtable.md` 後的下一棒開始前，管線暫停並出現竄改警告。
- [ ] 驗證席收到的指令內含 commit 清單與「越界即不通過」要求。

## 5. 管線狀態機

**啟動（▶ 開始執行，僅 HOST）前置檢查**，任一不過即拒絕啟動並在逐字稿說明：

1. 專案目錄是 git repo；
2. working tree 乾淨（無未提交變更）——避免 AI 變更與人類手動變更攪在一起、回滾誤傷；
3. 建立並切換到專屬分支 `roundtable/dev-<時間戳>`；若 `tasks.json` 存在未完成狀態（續作），改為切回其記錄的分支。

**主流程**（每一棒 = 一次席位 subprocess 呼叫）：

```text
拆任務（主控，讀完整逐字稿＋專案）
  → 逐任務迴圈：
      實作棒（agy）
        → 伺服器 git commit（無變更視為該次嘗試失敗，回實作棒）
        → 驗證棒（Codex，讀任務簡報＋commit diff）
            → 通過：任務標記 done，取下一任務
            → 不通過：驗證意見自動帶回實作棒重試
        → 同一任務失敗 3 次 → 仲裁棒（主控）：
            重派（附新指示，重試計數歸零）／跳過（標記 blocked）／詢問（暫停等主持人）
  → 任務清空或全數 blocked → 收尾棒（主控寫交接摘要進逐字稿）→ 管線結束
```

**上下文裁剪**：實作棒只讀「規格文件＋當前任務簡報＋上一次驗證意見」；驗證棒只讀「任務簡報＋本任務 commit diff」；只有主控的拆任務／仲裁／消化棒讀完整逐字稿。管線內每棒的上下文是固定小包裹，不隨會議長度膨脹。

**全域煞車**：

- 單任務重試上限：3 次（仲裁重派後歸零，同一任務最多仲裁 1 次，再失敗即 blocked）。
- 整場總棒數上限（預設 40，環境變數可調），達上限即自動暫停。
- **用量限制偵測**：席位呼叫失敗且輸出符合限流特徵（rate limit／usage limit／429 等關鍵字啟發式）時，視為「限流」而非「任務失敗」——不計入重試次數，管線立即暫停，狀態已在 `tasks.json`，稍後可續作。

**暫停與續作**：

- 「⏸ 暫停」按鈕：當前棒跑完後停下，狀態保留，按「▶」續跑。
- 人類插話：當前棒跑完後暫停 → 主控消化棒（讀新訊息，可輸出修訂後任務清單）→ 自動續跑。
- 伺服器重啟：啟動時發現 `tasks.json` 為執行中狀態，一律降為「已暫停（中斷）」，等 HOST 按「▶」續作。
- 管線執行或暫停中，「🧹 新會議」不可用（需先讓管線結束或接受封存告警——v1 直接禁用，訊息提示原因）。

### 驗收

- [ ] 髒 working tree／非 git repo 時按「▶」被拒且逐字稿有說明。
- [ ] 管線變更全部落在 `roundtable/dev-*` 分支，main 無任何新 commit。
- [ ] 驗證不通過的意見會出現在下一次實作棒的指令中。
- [ ] 同一任務第 3 次失敗後出現仲裁棒，三種裁決路徑行為正確。
- [ ] 達總棒數上限自動暫停；限流輸出觸發暫停且該任務重試計數不變。
- [ ] 插話後出現主控消化棒，且管線自動續跑；⏸ 後不會自動續跑。
- [ ] kill 伺服器再重啟，管線狀態為「已暫停」，按「▶」可從中斷任務續作。

## 6. 輸出協議（prompt 與解析器之間的逐字契約）

實作者不得更動下列格式字串；prompt 模板與解析器必須引用同一份常數。

**6.1 主控拆任務／消化棒**：輸出末尾必須含一個 ```json 圍欄區塊：

```json
{
  "tasks": [
    {
      "id": 1,
      "title": "建立 /api/login 端點",
      "files": ["api/login.py", "routes.py"],
      "acceptance": ["POST 正確帳密回傳 token", "錯誤帳密回傳 401"]
    }
  ]
}
```

- 伺服器取**最後一個** json 圍欄解析；圍欄前的自然語言照常入逐字稿。
- 消化棒輸出完整修訂清單；已 done 的任務由伺服器保留其狀態與 commit 紀錄，不得被重開。

**6.2 驗證棒**：輸出最後一行必須是（照抄格式）：

```text
驗證結果：通過
驗證結果：不通過—<一句話原因>
```

**6.3 仲裁棒**：輸出最後一行必須是：

```text
仲裁：重派—<給實作席的新指示>
仲裁：跳過—<原因>
仲裁：詢問—<要主持人拍板的問題>
```

**6.4 解析失敗處理**：任何協議棒解析失敗 → 原指令附加「上次輸出不符格式」重問一次 → 再失敗即暫停管線並在逐字稿說明，**不猜**。

### 驗收

- [ ] 三種協議的合法輸出都被正確解析並驅動對應狀態轉換。
- [ ] 不含協議標記的輸出觸發重問；連續兩次失敗觸發暫停。
- [ ] 消化棒無法將 done 任務改回未完成。

## 7. 資料與安全網

- **git commit 由伺服器執行**（實作棒 subprocess 結束後 `git add -A` + commit，訊息含任務編號、席位、嘗試次數），不依賴 AI 自行 commit。驗證席永遠有確定性的 diff 可查。
- **`tasks.json` 原子性寫入**：每次狀態轉換時寫暫存檔再 replace，任何時點的檔案內容都是完整合法 JSON。它是斷點續作的唯一事實來源。
- **稽核日誌**：每一棒的完整指令、stdout、stderr、exit code、耗時落地到 `data/devlogs/`，檔名可對應到任務與席位。逐字稿只放回覆本文，debug 靠日誌。
- **會議封存**：開新會議時 `tasks.json` 與逐字稿一起封存（既有封存機制的延伸）。

### 驗收

- [ ] 每個實作棒後專屬分支上恰有一個對應 commit，訊息格式正確。
- [ ] 管線任一時點強制結束伺服器，`tasks.json` 皆可被完整解析。
- [ ] `data/devlogs/` 內每棒一檔，內容含指令與完整輸出。

## 8. UI

- 頁首明顯的「開發模式」標示（讓人一眼知道這桌 AI 有筆）。
- 輸入區「送出」旁新增「▶ 開始執行」（僅 HOST、僅開發模式顯示）；管線執行中變為「⏸ 暫停」。
- 比照共識討論的 `discussBar`：管線執行中顯示狀態列（目前任務 m/n、哪一棒、分支名；暫停／警告狀態）。
- 管線訊息沿用現有氣泡，以副標籤標明「任務 n · 實作／驗證／仲裁」。

### 驗收

- [ ] 討論版看不到任何開發模式 UI 元素。
- [ ] 按鈕狀態（▶／⏸／不可用）與管線狀態一致。
- [ ] 狀態列在拆任務、實作、驗證、暫停、竄改警告各狀態下顯示正確資訊。

## 9. 已知限制（接受的殘餘風險）

1. **實作席（agy + bypass）可寫主機任何位置**。專案目錄外的寫入不在 git 也不在驗證席視野內，**無法偵測**，僅靠模型自律。§4 的兩道偵測只覆蓋專案目錄內越界與 `data/` 竄改。接受理由：自動開發無法逐次按權限；此模式僅限本機單人使用。
2. 限流偵測是關鍵字啟發式，可能漏判（誤計為任務失敗）或誤判（可續作，影響小）。
3. 「暫停」不砍正在執行的 subprocess——最長要等當前棒自然結束（受開發模式單棒 timeout 上限保護）。
4. 雜湊比對只能事後偵測竄改並停線，不能阻止竄改發生。
